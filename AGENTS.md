# Agent Rules

## Build and Test Verification (Repo-Specific)

ALWAYS verify that the package builds successfully and tests pass after making changes:

1. Run the package build: `yarn nx run @kirill.konshin/<package-name>:build`
2. Run the package tests: `yarn nx run @kirill.konshin/<package-name>:test`
3. You may use grep or similar filtering (e.g., `vitest run -t "pattern"`) to limit tests when you are confident the filter covers all relevant tests
4. If in doubt about test coverage, the final verification step should be to run all tests in the affected package

Do not consider work complete until build and tests pass.

# Releases

Automated with Nx Release + Conventional Commits (see `release` config in root `nx.json`):

- Commit type determines the bump: `fix` = patch, `feat` = minor, `!` or `BREAKING CHANGE:` footer = major, other types = no release
- A package is released when a releasable commit touches its files — the commit scope is informational only
- Versions in source `package.json` files are always `0.0.0`; real versions live in git tags (`@kirill.konshin/<pkg>@x.y.z`) and are applied only at publish time on CI

Current state: individual packages with independent versions. Used to be megapackage, now abandoned.

# Dev Dependencies

Repo is using hybrid NX-like dependency management strategy:

- [Single Version Policy](https://nx.dev/concepts/decisions/dependency-management#single-version-policy) for truly common packages like `react`, `typescript`, `wait-on` are defined in root `package.json`
- [Independently Maintained Dependencies](https://nx.dev/concepts/decisions/dependency-management#independently-maintained-dependencies) for all other packages
- Except demos, all dependencies are defined individually in their `package.json`

# Required Files Per Package

Each package MUST have these files with standardized content:

## `vite.config.ts`

```ts
import config from '../utils-private/vite.config';
export default config;
```

## `tsconfig.json`

```json
{
    "extends": "../utils-private/tsconfig.json",
    "compilerOptions": {
        "rootDir": "src",
        "outDir": "dist",
        "declarationDir": "dist"
    },
    "include": ["src"],
    "exclude": []
}
```

## `package.json` Scripts Section

All packages use identical scripts (copy exactly):

```json
{
    "scripts": {
        "----- BUILD -----": "",
        "clean": "rm -rf dist .tscache tsconfig.tsbuildinfo",
        "build": "vite build",
        "build:index": "cti create ./src",
        "build:check-types": "attw --pack .",
        "start": "yarn build --watch",
        "wait": "wait-on ./dist/index.js",
        "----- TEST -----": "",
        "test": "vitest run --coverage",
        "test:watch": "vitest watch --coverage",
        "----- STORYBOOK -----": "",
        "storybook:start": "storybook dev -p 6006",
        "storybook:build": "storybook build"
    }
}
```

Note: Use `"test:disabled"` instead of `"test"` if tests are not yet implemented.

## `package.json` Structure Rules

1. **Required fields**: `name`, `description`, `version`, `type: "module"`, `scripts`, `publishConfig`, `author`, `license`
2. **Exports format** (always use this exact structure):

```json
{
    "exports": {
        ".": {
            "import": {
                "import": "./dist/index.js",
                "types": "./dist/index.d.ts"
            }
        }
    },
    "main": "./dist/index.js",
    "module": "./dist/index.js",
    "types": "./dist/index.d.ts"
}
```

3. **DevDependencies**: Always include `"@kirill.konshin/utils-private": "*"`
4. **PeerDependencies**: Framework deps (react, next, electron, etc.) go in `peerDependencies` with `optional: true` in `peerDependenciesMeta`
5. **Dependencies**: Only runtime dependencies that consumers need

## Auto-Generated Fields (DO NOT manually edit)

The following `package.json` fields are auto-generated by `packages/utils-private/vite.exports.ts` during build via `fixExports()`:

- `exports` - generated from `src/*/` folder structure
- `main`, `module`, `types` - derived from main export
- `type` - set to `"module"`
- `publishConfig` - set to `{ "access": "public" }`
- `author`, `license` - standardized values
- `scripts.wait` - generated from main export path
- `peerDependenciesMeta` - auto-populated with `optional: true` for all `peerDependencies` (omitted when there are none)
- `peerDependencies` version ranges - normalized from `devDependencies` versions

**Only manually set**: `name`, `description`, `version`, `dependencies`, `devDependencies`, `peerDependencies` (keys only, versions are normalized)

# Creating a New Package

1. Copy an existing package folder (e.g., `packages/react`)
2. Update `name` in `package.json` to `@kirill.konshin/<package-name>`
3. Keep all config files (`vite.config.ts`, `tsconfig.json`) unchanged
4. Adjust dependencies as needed

# Nx Tasks

All common tasks are defined in `targetDefaults` in root `nx.json`: `clean`, `build`, `test`, `start` & `wait`. Task names are uniform across packages and demos (`build`, `start`, not `build:demo`). `build` depends on `^build`, so dependencies build first (cached in `.nx/cache`); demos override `build` in their `package.json` `nx` key to disable caching. `start` is a continuous task that depends on `^start` and `^wait`: dependency watchers start automatically, and `wait` (`wait-on ./dist/index.js`) blocks until the upstream watcher has actually emitted its dist — Nx starts a dependent of a continuous task as soon as the process spawns, without knowing whether it produced output, and `wait` avoids the redundant `^build` a fresh checkout would otherwise need before watching.

Root script convention: the bare name runs everything (`yarn build` / `test` / `start` / `serve` / `lint` cover packages + demos, with concurrency flags defined once there); suffixed variants narrow the scope by chaining the bare script — `xxx:packages` (publishable `@kirill.konshin/*` only), `xxx:affected` (affected vs `origin/main`, demos excluded). PR CI runs `lint`, `test:affected` and `build:affected`; the release workflow runs `lint`, `test` and `build:packages`.

# Module Boundaries

Every package is tagged in its `package.json` `nx` key with a `platform:*` tag (`universal`, `node`, `browser`, `worker`, `react`, `next`, `react-native`, `electron`, `tooling`, `umbrella`); demos are `type:demo`. The `@nx/enforce-module-boundaries` ESLint rule (configured in root `eslint.config.mjs`) restricts which platforms may import which — e.g. `platform:react` code cannot import `platform:worker` or `platform:node` packages. When creating a package, assign the appropriate tag; when a legitimate new dependency direction appears, extend `depConstraints` in `eslint.config.mjs` rather than removing tags.

Library packages run `start` (`vite build --watch`) which produces `dist/index.js`.
